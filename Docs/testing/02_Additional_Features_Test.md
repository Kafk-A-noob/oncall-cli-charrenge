# Phase 2: 追加実装機能 手動・結合検証テストケース

本ドキュメントは、コア機能完成後に追加実装された「堅牢化・デバッグ・UX向上」機能が、実際のE2E環境で正しく動作するかを証明するためのテスト結果です。

## 1. `--debug` フラグの動作検証

* **目的:** デバッグモードが正しく発火し、マスキング前後やLLMへの生データが可視化されるか検証する。
* **手順:** `oncall --debug Discord` または `oncall --debug ぎっとはぶ` を実行。
* **期待値:** コンソールに `[DEBUG] Function Calling 詳細` などの内部プロセスが詳細出力された後、最終レポートが出力されること。
* **結果 (Pass):**
  正常に内部動作のプロセスが出力されたことを確認。裏側でマスキングフィルターを通っていること、対象のAPIレスポンスの生データが取れていることが画面上で確認できた。

## 2. マルチバイト（ひらがな・カタカナ）の揺れ吸収検証

* **目的:** Zodの `transform` パイプラインが、日本語のゆらぎ入力を正しく規定のシステム名「Discord / GitHub / Cloudflare」に正規化し、APIエラーを防げるか検証する。
* **手順:** `oncall ぎっとはぶ` を実行。
* **期待値:** 「対象が見つかりません」等でエラーにならず、正しくGitHubのAPIが叩かれて結果が返ってくること。
* **結果 (Pass):**
  以下のように、ひらがな入力から「github」の引数に正規化され、正常に結果を出力することを確認。

```text
[思考中...] 状況を確認しています...
[実行] github のステータスを取得中...

--- [DEBUG] Function Calling 詳細 ---
  ツール名: fetchServiceStatus
  引数: {"serviceName":"github"}
  API応答(生): {"service":"GitHub","status":"All Systems Operational","indicator":"none"}
  マスク済み: {"service":"GitHub","status":"All Systems Operational","indicator":"none"}
--- [DEBUG] END ---

 【レポート】
GitHub: 全てのシステムは正常に稼働しています。
```

## 3. 無限ループ防止（フェイルセーフ）の動作検証

* **目的:** LLMのエラー（ハルシネーションによる無限のツール呼び出し）等によってAPI呼び出しがループし、APIの過剰な課金が発生するのを防ぐ上限機構（5回）が機能するか。
* **手順:** コードによる静的解析および意図的なループテスト。
* **期待値:** 呼び出し回数が5回を超えた段階で強制的にループを終了すること。
* **結果 (Pass):**
  `agent.ts` の `while (response.functionCalls && response.functionCalls.length > 0)` ブロック内に `if (loopCount >= 5) break;` のフェイルセーフ機構が存在し、意図しない課金バーストをプログラムレベルでブロックできていることを確認。

## 4. API無料枠超過（429 Error）の例外処理検証

* **目的:** LLMのAPIリミット（Quota）到達時に、アプリがクラッシュせずにセーフティネットの例外ルートを通り、ユーザーフレンドリーなエラーを返すか。
* **手順:** 意図的に多数のリクエストを送信し、APIの無料枠を枯渇させた状態でCLIを実行する。
* **期待値:** スタックトレース（生のエラー）を吐かずに安全に終了すること。
* **結果 (Pass):**
  開発中にAPI利用制限に到達した際、以下の制御された日本語のエラーメッセージが正しく標準エラー出力に吐き出され、ツールがフリーズせずに終了したことを確認済。
  `[エラー] Gemini APIの無料枠(Quota)上限に到達しました。しばらく時間をおいてから再度お試しください。`
