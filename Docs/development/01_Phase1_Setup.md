# Phase 1: プロジェクト環境構築 (Environment Setup)

## 概要

今後のフェーズで必要となる Bun + TypeScript (Strict) の実行基盤と、LLM連携（`@google/genai`）、強固なバリデーション（`zod`）の土台を作成します。
こちらの手順に従い、ユーザー様ご自身でターミナルおよびエディタにて以下の作業を実行してください。

## 作業手順

### 1. パッケージの初期化とインストール

ターミナルを開き、対象ディレクトリ (`oncall-cli-charrenge`) 上で以下のコマンドを順番に実行してください。

```bash
# 0. 納品用リポジトリディレクトリへ移動する (超重要)
cd oncall-cli-charrenge

# 1. プロジェクトの初期化（package.json等が生成されます）
bun init -y

# 2. アプリケーションに必要な中核パッケージのインストール
bun add @google/genai zod

# 3. TypeScript開発環境の構築
bun add -d typescript @types/node
```

> **なぜこれが必要か:**
> 制約事項である Bun のランタイム環境を確立するとともに、コアロジックで利用する Gemini 用 SDK と Zod を導入するためです。

---

### 2. `package.json` の修正

インストール完了後、生成された `package.json` を開き、CLIを即座に起動できるよう `scripts` を追記してください。

**修正対象:** `package.json`
**追加内容:** `"start": "bun run src/cli/index.ts"`

```json
  "scripts": {
    "start": "bun run src/cli/index.ts"
  },
```

*(※上記を既存のJSONオブジェクトに追加してください)*

> **なぜこれが必要か（詳細解説）:**
> `package.json` 内の `scripts` プロパティは、プロジェクト固有のコマンドに「短い別名（ショートカット）」を定義するための機能です。
>
> **1. グローバル環境への影響について:**
> ここに追記しても、PC全体（グローバル環境）にコマンドがインストールされるシステム汚染は**一切起きません**。あくまでこのプロジェクト・フォルダ内で `bun run` と打った時だけ動く「局所的なエイリアス」です。
>
> **2. `start` コマンドを設定する目的:**
>
> - **パスの隠蔽:** プロジェクトのエントリーポイント（実行の起点）が `src/cli/index.ts` の奥深くにあっても、毎回そのフルパスを打つ必要がなくなります。
> - **共通のインターフェース:** Node.jsやBunのエコシステムでは、「開発したアプリを動かすための統一ルール」として `start` という名前が慣習になっています。
> この記述をしておくことで、中身のディレクトリ構成を知らない第三者（選考官など）でも、「とりあえず `bun run start` を実行すればアプリが立ち上がる」と迷わずに操作できるようになるため、CLIツールにおいて非常に重要です。

---

### 3. 関心の分離に基づくディレクトリ・ファイルの作成

アーキテクチャの要件に従い、UIとコアロジックを分離するためのフォルダとファイル（中身は空で構いません）を作成してください。

1. `src/cli/index.ts`  (CLIのエントリーポイント)
2. `src/core/agent.ts`  (LLM推論等のコアロジック)
3. `src/core/masking.ts`  (機密情報の置換処理)
4. `src/tools/index.ts`  (APIリクエスト等の外部関数用)

> **なぜこれが必要か:**
> MVP時から密結合を防ぎ、将来的な Web API (Hono) への拡張時に `core` や `tools` をそのまま流用できるようにするためです。
