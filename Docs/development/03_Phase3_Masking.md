# Phase 3: データマスキング処理の実装

## 概要

システムで取得した情報を外部の生成AI（LLM）へ送信する際、もっとも警戒すべきなのが**機密情報（IPアドレスや各種トークン、個人情報）の漏洩**です。

本フェーズでは、「LLMの推論エンジンへデータを渡す直前に必ず通す関所」として、正規表現を使ったマスキング機能（フィルター）を `src/core/masking.ts` に実装します。

## 作業手順

### `src/core/masking.ts` の実装

エディタで `src/core/masking.ts` を開き、以下のコードを実装してください。

```typescript
/**
 * セキュリティを担保するためのデータマスキング処理。
 * 外部APIのレスポンスやログ文字列から、機密情報らしきパターンを検出し
 * 強制的に [MASKED] へ置換します。
 */

const MASK_REPLACEMENT = "[MASKED]";

// 検出のための正規表現（Regex）定義
// MVPの要件に基づき、IPv4アドレスやクレデンシャル等を対象とします。
const SENSITIVE_PATTERNS = [
  // 1. IPv4アドレスの検出 (例: 192.168.1.1)
  /\b(?:\d{1,3}\.){3}\d{1,3}\b/g,


  // 2. IPv6アドレスの検出 (例: 2001:0db8::8a2e:0370:7334 や ::1 など)

/*初期のコード
/(?:(?:[a-fA-F0-9]{1,4}:){7}[a-fA-F0-9]{1,4}|(?:[a-fA-F0-9]{1,4}:){1,7}:|:(?::[a-fA-F0-9]{1,4}){1,7}|(?:[a-fA-F0-9]{1,4}:){1,6}:[a-fA-F0-9]{1,4})/g,
*/

  /*
    2. IPv6アドレスの検出 (例: 2001:0db8::8a2e:0370:7334 や ::1 など)
    初期の記述では見づらかったため、可読性と保守性向上を狙いnew RegExpと配列結合を使用
  */

  // ※可読性と保守性向上のため、new RegExp と配列結合を使用
  new RegExp([
    '(?:',
      '(?:[a-fA-F0-9]{1,4}:){7}[a-fA-F0-9]{1,4}',     // 完全表記
      '|(?:[a-fA-F0-9]{1,4}:){1,7}:',                 // 後方省略 (例: 2001:db8::)
      '|:(?::[a-fA-F0-9]{1,4}){1,7}',                 // 前方省略 (例: ::1)
      '|(?:[a-fA-F0-9]{1,4}:){1,6}:[a-fA-F0-9]{1,4}', // 中間省略 (例: 2001::1)
    ')'
  ].join(''), 'g'),
  
  // 3. Bearer または Token に続くクレデンシャル文字列（APIキー漏洩対策）
  /(?:Bearer|Token|token)\s+[A-Za-z0-9\-\._~]+/g,

  // 4. メールアドレス等の検出 (例: admin@example.com)
  /\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b/g,

  /*
    5. JWT (JSON Web Token) の検出 (例: eyJhbG... から始まる文字列)
    認証エラー時のログ漏洩を防いでくれる
  */
  /eyJ[A-Za-z0-9-_]+\.[A-Za-z0-9-_]+\.[A-Za-z0-9-_]+/g,

  /* 
    6. URL内のBasic認証情報 (例: https://user:pass@example.com の user:pass 部分)
    通信エラーログ等でのパスワード漏洩を防ぐ。
  */
  /(?<=https?:\/\/)[^:\/]+:[^@\/]+(?=@)/g,

];

export function maskSensitiveData(input: string): string {
  if (!input) return input;
  
  let maskedText = input;

  // 定義したパターンを全て検査し、見つかれば置換する
  for (const pattern of SENSITIVE_PATTERNS) {
    maskedText = maskedText.replace(pattern, () => {
      // 今回は安全性を最優先し、マッチした箇所を一律でマスキングする
      return MASK_REPLACEMENT;
    });
  }

  return maskedText;
}
```

### この実装における技術的な意図（シニアエンジニア解説）

* **関心の分離を生かしたゲートウェイ設計:**
    APIを叩く処理（`Phase 2`）の中にマスク処理を混ぜ込まないことが最大のポイントです。マスキングロジックを独立した `masking.ts` に切り出しておくことで、「今後新しくAWSやGCP用のツール関数などを追加したとしても、LLMに渡す手前で絶対にこの `masking.ts` 関数を通す」という確固たるルールを敷くことができ、セキュリティ監査(Audit)が非常に簡単になります。
* **柔軟なフィルターの拡張性:**
    `SENSITIVE_PATTERNS` という配列で管理することで、後から「JWTトークンも消したい」「社内の特定のドメイン情報を消したい」となった際に、正規表現を1行追加するだけで対応できる保守性の高い設計となっています。
